# Technical Decisions — Dakota Analytics Energy Pipeline

---

## 1. Orchestration: Dagster

Dagster was chosen over Airflow and Prefect because it models data assets rather than tasks. Every stage in this pipeline — raw EIA tables, cleaned Silver models, Gold KPIs, and the final report is a tracked asset with its own materialisation history and row-count metadata.

The practical benefits for this pipeline:
- The full lineage from raw ingestion to report is visible in the Dagster UI without any custom instrumentation
- `dagster-dbt` promotes every dbt model to a Dagster asset, so dbt runs are not a black box
- Both scheduled runs (EIA daily batch) and sensor-driven runs (enrichment API) are managed in the same place
- Only two containers are needed: webserver and daemon.

Airflow requires four separate processes and treats dbt as an external operator with no lineage integration. Prefect has similar limitations on lineage tracking.

---

## 2. Database: PostgreSQL 15

PostgreSQL was used across two databases in the same instance: `energy_analytics` for pipeline data and `dagster_db` for Dagster's own run history.

The three-schema layout maps directly to the Medallion layers:
- `raw` — Bronze ingestion tables
- `staging` — Silver dbt models
- `analytics` — Gold dbt models

PostgreSQL's `ON CONFLICT DO NOTHING` handles EIA idempotency cleanly. The same database also supports Dagster's state storage via `dagster-postgres`.

---

## 3. dbt Architecture: Medallion (Bronze → Silver → Gold)

Medallion was chosen over others because this pipeline has two sources with different structures and update frequencies. EIA publishes monthly, the enrichment service runs on demand. A star schema requires stable, well-defined dimensions upfront. Medallion separates concerns by layer and makes it straightforward to add new sources later.

- **Bronze** — views directly over raw tables. No transformation, just a stable reference.
- **Silver** — incremental models that clean and normalise. Only new rows are processed on each run, which keeps transformation time predictable as data grows.
- **Gold** — full table refresh on every run. Four models: generation mix, price trends, carbon footprint, and executive summary.

The Silver incremental strategy uses `merge` on natural keys. If source data changes shape, a `dbt run --full-refresh` rebuilds from scratch.

dbt tests are defined on all critical columns. Two range tests on the Gold generation model are set to `warn` severity rather than `error`, because EIA data occasionally contains negative net generation values (energy imports) which are valid data points.

---

## 4. API: FastAPI

FastAPI was used for the enrichment service because it produces automatic Swagger documentation at `/docs`, enforces request and response shapes through Pydantic, and starts up quickly in a container. The three domains: weather, carbon, and market — each have their own router file, keeping the codebase easy to navigate.

Pydantic v2 is used throughout for data validation, which is consistent with the ingestion layer and orchestration layer that also use Pydantic models.

---

## 5. Ingestion: httpx + tenacity

`httpx` handles both the EIA API and enrichment API calls asynchronously. The enrichment client fetches all nine grid regions concurrently rather than sequentially, which keeps run time low.

`tenacity` provides retry logic with exponential backoff. It retries on 5xx responses and 429 rate limits, logs each retry attempt to the structured log, and stops after three attempts. This makes transient API failures visible in Dagster's event log without crashing the run.

All writes go through SQLAlchemy with a connection pool. Every ingestion run writes a record to `raw.ingestion_audit_log`, which appears in the Pipeline Data Quality section of the report.

---

## 6. Reporting: Jupyter to HTML

The report is generated by building a Jupyter notebook programmatically with `nbformat`, executing it with `nbconvert`'s `ExecutePreprocessor`, and rendering it to a standalone HTML file. The HTML output has code cells hidden, the viewer sees only charts and tables.

Plotly charts are rendered as HTML strings using `fig.to_html()` before being displayed in the notebook. This ensures nbconvert can embed them in the output rather than skipping them as unrecognised MIME types.

The report covers six sections: executive KPI summary, generation mix, price analysis, carbon footprint, regional market performance, and ingestion audit log. It reads directly from the Gold tables in PostgreSQL at generation time.

---

## 7. Dependency Management: uv

`uv` is used in all three Dockerfiles for installing Python dependencies. It is significantly faster than pip for resolving and installing packages, which reduces Docker build time. It is a drop-in replacement for `pip install` and works with standard `pyproject.toml` files without requiring a separate lock file format.
